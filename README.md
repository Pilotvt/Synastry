# Synastry Desktop

Synastry — это десктопное приложение на Electron + React для расчёта натальных карт, синастрии и офлайн-исследований на основе реальных созвездий IAU. Python-бэкенд поставляется вместе с инсталлятором, поэтому пользователю достаточно установить один пакет.

## Обзор автообновления
- GitHub Releases хранит все публичные сборки. Каждая публикация содержит подписанный NSIS-установщик, переносимый ZIP и манифест `latest.yml`, который генерирует electron-builder.
- `electron-updater` настроен на провайдера `github`. После запуска упакованная версия приложения молча вызывает `autoUpdater.checkForUpdates()`, получает метаданные релиза и спрашивает пользователя перед загрузкой.
- Все события жизненного цикла апдейта пробрасываются в рендерер через каналы `updates:status` / `updates:error`, чтобы UI мог показывать прогресс или ошибки.
- Ошибки фиксируются в `logs/main.log` (папка `%APPDATA%/Synastry/logs`). Пользователь видит диалог «не удалось проверить обновления, попробуйте позже».

### Артефакты релиза
Каждый запуск `npm run dist` складывает сборку в каталог `release/`:

1. `Synastry-<version>-setup.exe` — NSIS-установщик, который использует автообновление.
2. `Synastry-<version>-win.zip` — переносимая сборка для ручной установки и диагностики.
3. `latest.yml` — манифест с ссылками и SHA512, который читает `autoUpdater`.

Эти три файла нужно прикрепить к GitHub Release с тегом, совпадающим с версией в package.json. Не переименовывайте артефакты после сборки, иначе `latest.yml` потеряет корректные ссылки.

### Процесс релиза
1. Убедитесь, что зависимости установлены и линтер проходит без ошибок.
2. Соберите фронтенд, подготовьте бэкенд и сформируйте релизные артефакты:

   ```pwsh
   npm install
   npm run backend:setup
   npm run dist
   ```

3. Создайте черновик GitHub Release с тегом вроде `v1.0.0-beta`, приложите `Synastry-<version>-setup.exe`, `Synastry-<version>-win.zip` и `latest.yml`.
4. Опубликуйте релиз — при следующем запуске клиенты автоматически его увидят.

### Требования к окружению
- Для публичных репозиториев никакой токен не нужен. Для приватных задайте токен с правом `repo` в переменную `GH_TOKEN` (или `GITHUB_TOKEN`) перед запуском electron-builder.
- Пример для PowerShell:

  ```pwsh
  $Env:GH_TOKEN = '<github-personal-access-token>'
  npm run dist
  ```

## Поведение клиента
- При запуске основной процесс настраивает логирование, прогоняет миграции, стартует встроенный Python-сервис и только после этого проверяет обновления (только в упакованной версии).
- Если апдейт найден, показывается модальное окно с описанием версии и релиз-нотами из GitHub; загрузка начинается только после согласия пользователя.
- После скачивания приложение предлагает «Перезапустить и установить» и вызывает `autoUpdater.quitAndInstall()`.
- Ошибки загрузки логируются и отображаются в диалоге/IPC.

### Ручная проверка обновлений
- В меню «Справка» есть пункт «Проверить обновления» (`Ctrl+Shift+U`), который вызывает `autoUpdater.checkForUpdates()` и показывает результат в диалогах.
- Рендерер может вызвать тот же сценарий через `ipcRenderer.invoke('updates:check-now')`, если нужен собственный UI прогресса.

### Логирование и телеметрия
- Весь апдейт-лог попадает в `logs/main.log` (electron-log). Этот файл используют при разборе инцидентов у пользователей.

## Миграции данных
- Основной процесс сравнивает `app.getVersion()` с записью в `%APPDATA%/Synastry/migration-info.json`.
- Если версия изменилась, вызывается `performDataMigrations(prevVersion, nextVersion)` ещё до загрузки React-бандла. Добавляйте туда преобразования кэшей, файлов и т.п.

### Как добавить миграцию
1. Откройте `electron/main.js` и найдите `performDataMigrations`.
2. Добавьте нужные шаги (перепаковка JSON, изменение схемы SQLite и т.д.).
3. Держите код идемпотентным, чтобы пользователи могли перепрыгивать через несколько версий.
4. После успешного выполнения обновите версию в `migration-info.json`, чтобы миграция больше не выполнялась.

## Полезные команды
- `npm run desktop:dev` — сборка фронтенда и запуск Electron поверх исходников.
- `npm run lint` — проверка ESLint.
- `npm run dist` — полный цикл сборки, упаковки Python и выпуск подписанных инсталляторов + манифестов в `release/`.

## API модерации
- **Текст**: `POST /api/moderation/text` с телом `{ "text": "...", "language_hint": "ru" }`. Используются `better-profanity` и fastText. В ответе приходят `matches`, `reasons` (словари, транслит, fastText) и `censored_text`, чтобы UI сразу показывал причину блокировки.
- **Изображения**: `POST /api/moderation/image` (multipart, поле `file`). Python-прослойка прогоняет файл через NudeNet v3 detector (`nudenet`, `320n.onnx`), возвращает `label` (`safe|unsafe`), `raw_scores`, `confidence` и флаг `is_explicit` при `unsafe ≥ 0.6`. Вес детектора уже в репо (`app/nudenet/320n.onnx`) и при первом запуске копируется в `%USERPROFILE%/.NudeNet/` — интернет не нужен.
- **Расширение словаря**: добавляйте новые формы в `data/profanity_ru_ua.txt` (UTF‑8, по одному слову на строку). После перезапуска сервера изменения подхватываются автоматически.
- **Обработка ошибок**: пустые загрузки дают HTTP 400, проблемы с NudeNet — HTTP 502. При желании рендерер может перейти на облачный API, если сервер вернул ошибку.

### Упакованные активы
- `npm run backend:setup` собирает окружение (`python-env`), ставит зависимости и прогревает fastText-словарь (`scripts/build_backend_env.py`). NudeNet вес (`app/nudenet/320n.onnx`) уже лежит в репозитории, ничего не качаем.
- За транслитерацию и маскированные ругательства отвечает лёгкий fastText-классификатор (`fasttext-wheel`). Корпус собирается из `data/profanity_ru_ua.txt` (кириллица + латиница) и набора нейтральных фраз. Итоговая модель лежит в `data/profanity_fasttext.bin`, а `scripts/build_backend_env.py` пересобирает её автоматически во время подготовки окружения.
